# 虚拟列表图解

![虚拟列表图解](/img/virtual-list-diagram.png)

虚拟列表解决的是一个很具体的问题：数据很多，但用户同一时间只能看到屏幕里的一小段。如果把 1 万条、10 万条数据全部渲染成 DOM，页面会在创建节点、布局计算、绘制和内存占用上付出很大成本。

虚拟列表的思路是：**滚动条看起来像完整列表，真实 DOM 只保留可视区域附近的少量节点**。

### 1. 为什么需要虚拟列表

普通列表会把所有数据都渲染出来：

```text
100000 条数据 -> 100000 个 DOM 节点
```

这会带来几个问题：

- 首屏渲染慢。
- DOM 节点太多，布局和绘制成本高。
- 滚动时容易卡顿。
- 内存占用持续升高。

虚拟列表把真实渲染量控制在一个小窗口里：

```text
100000 条数据 -> 可视区 20 条 + 缓冲区若干条
```

用户滚动时，列表项会被复用或重新计算位置，看起来像完整列表，实际页面上只有很少的 DOM。

### 2. 固定高度虚拟列表的核心公式

固定高度场景最容易理解。假设每一项高度都是 `itemHeight`，容器滚动距离是 `scrollTop`。

开始渲染下标：

```js
const startIndex = Math.floor(scrollTop / itemHeight);
```

可视区能容纳的数量：

```js
const visibleCount = Math.ceil(containerHeight / itemHeight);
```

结束下标：

```js
const endIndex = startIndex + visibleCount;
```

为了避免快速滚动时出现白屏，通常会加一段缓冲区：

```js
const overscan = 5;
const renderStart = Math.max(0, startIndex - overscan);
const renderEnd = Math.min(list.length, endIndex + overscan);
```

真实渲染的数据就是：

```js
const visibleList = list.slice(renderStart, renderEnd);
```

### 3. 为什么滚动条仍然正常

虚拟列表虽然只渲染少量 DOM，但必须让滚动条表现得像完整列表。

做法是创建一个占位层：

```js
const totalHeight = list.length * itemHeight;
```

然后让真实渲染的列表窗口移动到正确位置：

```js
const offsetY = renderStart * itemHeight;
```

对应到页面结构，大致是这样：

```html
<div class="scroll-container">
  <div class="spacer" style="height: totalHeight">
    <div class="visible-list" style="transform: translateY(offsetY)">
      <!-- 只渲染可视区附近的数据 -->
    </div>
  </div>
</div>
```

这就是虚拟列表的关键：**用一个很高的占位层撑开滚动条，用一个很小的真实列表负责展示内容**。

### 4. 滚动时发生了什么

一次滚动大致会经历这些步骤：

1. 用户滚动容器，产生新的 `scrollTop`。
2. 根据 `scrollTop` 计算 `startIndex`。
3. 根据容器高度计算 `endIndex`。
4. 加上 `overscan` 得到真实渲染范围。
5. 更新 `visibleList`。
6. 用 `translateY` 把这段列表移动到视觉上的正确位置。

如果实现得好，滚动过程中 DOM 数量基本稳定，只是内容和偏移量在变化。

### 5. overscan 的作用

`overscan` 指的是在可视区上方和下方额外渲染一部分数据。

它的作用是提升滚动体验：

- 快速滚动时减少白屏。
- 给浏览器一点提前渲染空间。
- 避免边界处频繁增删节点。

但 `overscan` 不是越大越好。太小容易白屏，太大又会增加 DOM 数量。实际项目里通常需要根据列表项复杂度、滚动速度和设备性能调节。

### 6. 动态高度列表更复杂

固定高度列表只需要一个简单公式：

```text
index = scrollTop / itemHeight
```

但动态高度列表里，每一项高度都可能不同，就不能直接除以固定高度了。

常见做法是维护一个高度缓存：

```js
const heightMap = new Map();
```

再维护每一项之前的累计高度，也就是前缀和：

```text
item 0: top = 0
item 1: top = height[0]
item 2: top = height[0] + height[1]
item 3: top = height[0] + height[1] + height[2]
```

滚动时需要根据 `scrollTop` 找到第一个可见项。数据量很大时，通常会用二分查找来定位，而不是从头遍历。

动态高度虚拟列表的难点主要在三处：

- 首次渲染前不知道真实高度，需要预估。
- 渲染后要测量真实高度并更新缓存。
- 高度变化后要修正后续项目的位置和总高度。

### 7. 工程实现注意点

虚拟列表看起来是渲染优化，但实际更像滚动系统工程。

几个容易踩坑的点：

- 滚动事件不要做太重的同步计算。
- 列表项组件要尽量轻，避免每次滚动触发大面积重渲染。
- 使用 `transform` 移动可视窗口，少改会触发布局的属性。
- 图片、富文本、折叠面板会让动态高度更难维护。
- 需要考虑键盘访问、焦点管理和无障碍语义。
- 如果有选中、展开、编辑态，状态要按数据 id 存，不要按当前渲染下标存。

### 8. 什么时候不该用虚拟列表

虚拟列表不是所有列表的默认答案。

如果列表只有几十项，普通渲染更简单，也更稳定。如果列表项高度复杂、交互密集、SEO 要求强，虚拟列表可能引入额外复杂度。

更适合使用虚拟列表的场景：

- 长表格。
- 聊天记录。
- 日志列表。
- 搜索结果页。
- 大型选择器。
- 后台系统中的海量数据列表。

核心判断标准是：**DOM 数量是否已经成为性能瓶颈**。

### 9. 一句话总结

虚拟列表的本质不是“少加载数据”，而是“少渲染 DOM”。

它通过总高度占位保证滚动体验，通过可视窗口控制真实渲染量，通过偏移量把少量节点放到正确位置。固定高度场景可以用简单公式解决，动态高度场景则需要高度缓存、测量和查找结构配合。
