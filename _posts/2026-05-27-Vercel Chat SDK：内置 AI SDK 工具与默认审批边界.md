# Vercel Chat SDK：内置 AI SDK 工具与默认审批边界

## 背景

做“能在 Slack/Teams/自有站点里跑起来的聊天机器人”时，工程复杂度往往不在 LLM 调用，而在：

- 消息读取/发送、线程/频道/权限等 **平台动作** 的封装；
- 把这些动作暴露成 **可控、可审计的工具（tools）** 给 agent；
- 把“读”和“写”能力严格隔离，并在默认情况下对写操作做审批（HITL）。

Vercel 在 2026-05-20 的更新里，把 Chat SDK 的读写动作以 **AI SDK 工具集** 的方式内置到 `chat/ai` 子路径：通过一次 `createChatTools(chat)` 即可把 Chat SDK 的能力接入到 AI SDK 的 agent/tool loop 体系里，并提供 `reader` / `messenger` / `moderator` 三种 preset 来约束工具范围，同时把写操作默认放在可审批边界内。

## 为什么重要

- **“业务工具”与“平台动作”终于能用同一套 agent 工具机制管理**：统一模型调用、工具调用、审批、观测与调试的工程路径。
- **默认安全姿势更合理**：写操作可以被 `requireApproval` 门控，降低“模型误操作”的风险。
- **更低的接入成本**：不再需要你手写一大堆把 Chat SDK 动作包装成 AI SDK tools 的胶水代码。

## 前端/AI 开发落地建议

### 1. 先从 `reader` 起步，再逐步放开到 `messenger`

把机器人能力分阶段上线：

- `reader`：仅允许读取与分析（例如摘要、检索、分类、意图识别）
- `messenger`：允许回复/转发/提醒等写动作，但对关键场景强制审批
- `moderator`：涉及管理/治理动作（比如内容审核、移除内容/成员）要更严格的权限与审计

### 2. 把审批当作产品体验的一部分

对需要审批的写工具，建议把“审批 UI”设计成可追踪的工作流，而不是一个临时弹窗：

- 在前端展示工具调用草案（将发送的内容、目标频道/用户、理由）
- 支持一键修改后再批准（人机协作编辑）
- 审批记录写入日志（谁批准、批准了什么、模型输入是什么）

### 3. 工具最小化 + 结构化上下文输入

避免把平台的所有能力都丢给 agent：

- 只暴露完成当前任务所需的工具集合
- 上下文（频道信息、用户信息、权限信息）尽量结构化注入，减少模型“猜测”
- 对涉及隐私/敏感数据的读工具增加范围限制（只读当前线程、只读最近 N 条等）

## 注意事项

- **权限与身份边界**：机器人代表谁行动（bot token vs user token）需要在系统层明确，否则很容易出现“越权写入/读取”。
- **幂等与重放**：聊天平台常见重试/重复事件；写工具必须做幂等处理，避免重复发消息/重复操作。
- **可观测性**：把工具调用、审批、最终写入动作串成一条 trace，便于追责与排障。
