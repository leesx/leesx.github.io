# Prompt API Polyfill：把浏览器原生 AI 接口带到全平台

## 背景

Chrome for Developers 在 2026-05-14 发布了 `An experimental polyfill for the Prompt API`。这篇内容解决的不是“浏览器里能不能跑 Prompt API”，而是一个更现实的问题：**当 Prompt API 只在部分浏览器和部分操作系统可用时，前端团队怎么避免把产品做成平台碎片化体验。**

官方给出的思路很直接：提供一个 **符合规范的 Prompt API polyfill**。在支持 `window.LanguageModel` 的环境下继续走浏览器原生实现；在不支持的环境下，则用云端模型或本地 `Transformers.js` 后端补齐同一套接口。

这意味着团队可以优先围绕平台 API 设计，而不是一开始就把业务层和某一家模型服务 SDK 强耦合。

## 为什么重要

- **前端终于可以围绕“标准接口”而不是“厂商 SDK”写 AI 能力**：这会直接影响组件抽象、测试方式和后续迁移成本。
- **浏览器内置 AI 开始具备渐进增强路径**：支持 Prompt API 的设备走原生能力，不支持的设备回退到 polyfill，产品形态不会因此裂开。
- **客户端 AI 的工程边界更清楚了**：什么时候保留本地处理，什么时候退回云端，什么时候要提醒用户离线不可用，官方都给了明确框架。

## 前端/AI 开发落地建议

### 1. 把 Prompt API 当成能力接口，而不是把 polyfill 当成主实现

推荐的工程策略是：

- 业务代码只面向 `LanguageModel` 这类浏览器接口
- 运行时先做能力检测
- 仅在缺失时动态加载 polyfill

这样做的好处是，未来浏览器原生支持扩大后，你不需要重写业务逻辑，只是让 fallback 的触发频率下降。

### 2. 明确区分“本地后端”和“云端后端”

官方文档把两条路径讲得很清楚：

- **云端后端**：不必等待本地模型可用，但会引入延迟、费用和隐私边界问题
- **本地后端**：数据保留在浏览器内，但需要模型下载、设备算力和兼容性兜底

这意味着前端产品不该只做一个统一按钮，而应该把能力策略体现在体验层：

- 对敏感文本优先走本地
- 对长上下文、多模态或高质量生成允许切到云端
- 对离线场景显式展示能力降级

### 3. 把密钥保护和成本控制前置到接入阶段

这篇文档最容易被忽略的一点，是它反复提醒你：**不要把 Gemini 或 OpenAI API key 直接暴露在 Web App 里。**

如果你要用云端后端，至少要同步设计：

- 服务端代理或安全网关
- 前端的调用限额与频率控制
- 成本可视化与超限降级

否则 polyfill 虽然能跑，但会把安全和计费风险直接抛给前端页面。

## 注意事项

- **polyfill 不等于原生语义完全一致**：走云端时，你失去了浏览器内置 AI 的本地处理、离线可用和共享模型缓存等优势。
- **多模态能力存在后端差异**：官方明确写到，不同后端的 structured output 和音视频输入支持并不完全一致。
- **不要把“兼容更多平台”误解成“可以忽略能力检测”**：仍然要根据浏览器、设备、网络状态和后端模式决定最终路径。
