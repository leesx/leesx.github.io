# Next.js 16：Cache Components 和 DevTools MCP

Next.js 16 在 2025 年 10 月发布，重点是 Turbopack、缓存模型、架构边界和 AI 辅助调试。其中最值得前端团队认真理解的是 Cache Components 和 DevTools MCP。

App Router 之后，Next.js 的缓存一直是强能力，也是不少 bug 的来源。Next.js 16 试图把缓存语义变得更显式：通过 Cache Components、`use cache` 和 Partial Pre-Rendering，把静态、缓存和动态内容放在同一个路由里组合。这样可以让页面先返回静态壳，再逐步填充动态区域。

对业务开发来说，重点不是记 API 名字，而是重新思考页面切分：

- 哪些内容可以长时间缓存？
- 哪些内容跟用户身份有关？
- 哪些内容必须每次请求都新鲜？
- 哪些 fallback 会影响首屏体验？

另一个变化是 Next.js DevTools MCP。MCP 让 AI 工具能拿到 Next.js 应用的路由、日志、错误和页面上下文。以前让 AI 帮忙排查问题，开发者要复制报错、贴目录、解释路由结构。DevTools MCP 的目标是让工具直接理解开发环境。

`middleware.ts` 改为 `proxy.ts` 也很重要。这个命名强调它是网络边界逻辑，不是普通中间件。做权限、重定向、国际化路由时，应该更谨慎地控制它的职责。

### 升级关注点

- Turbopack 成为默认路径后，检查插件和构建差异。
- 缓存逻辑需要显式设计，不要依赖旧版本的隐式行为。
- `middleware.ts` 迁移到 `proxy.ts` 时同步更新团队文档。
- AI 调试能力可用，但生产问题仍要有日志和监控闭环。
