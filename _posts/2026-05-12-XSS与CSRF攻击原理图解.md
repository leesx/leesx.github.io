# XSS 与 CSRF 攻击原理图解

![XSS 原理图解](/img/xss-principle-diagram.png)

![CSRF 原理图解](/img/csrf-principle-diagram.png)

前端安全里最常见、也最容易混淆的两个概念是 XSS 和 CSRF。

简单区分：

- XSS 关注的是“攻击者能不能把恶意脚本注入到页面里执行”。
- CSRF 关注的是“攻击者能不能借用用户已登录态发起非本人意愿的请求”。

它们都可能造成账号、资金、数据和权限风险，但攻击入口、防护手段和排查方式完全不同。

### 1. XSS 是什么

XSS，全称 Cross-Site Scripting，中文常叫跨站脚本攻击。它的核心是：攻击者把恶意脚本插入到页面中，当用户访问页面时，脚本在用户浏览器里执行。

一旦脚本执行成功，攻击者可能做这些事情：

- 读取页面里的敏感信息。
- 窃取 token、用户信息或业务数据。
- 劫持用户操作。
- 发起伪造请求。
- 篡改页面内容，诱导用户继续操作。

XSS 的关键不是“跨站”，而是“脚本在受害者浏览器里执行”。

### 2. XSS 的常见类型

#### 反射型 XSS

恶意代码来自 URL、表单或请求参数，服务端没有正确转义，直接把它返回到页面。

典型场景：

```text
攻击链接 -> 用户点击 -> 服务端返回带恶意脚本的页面 -> 浏览器执行脚本
```

这类攻击通常依赖用户点击特制链接。

#### 存储型 XSS

恶意内容被保存到数据库或后台系统里，其他用户访问页面时自动触发。

典型场景：

```text
攻击者提交恶意评论
-> 服务端保存
-> 其他用户查看评论
-> 恶意脚本执行
```

存储型 XSS 危害更大，因为它不需要每次诱导用户点击链接。

#### DOM 型 XSS

漏洞出现在前端代码中。页面从 URL、hash、localStorage 等位置读取数据，并通过危险 DOM API 写入页面。

危险写法：

```js
document.body.innerHTML = location.hash;
```

如果 hash 里包含恶意 HTML 或脚本，就可能触发 XSS。

### 3. XSS 的防护思路

XSS 防护的核心是：**不信任任何输入，输出到不同上下文前必须做正确处理。**

常见措施：

- 对 HTML 文本做转义。
- 对 URL、属性、JavaScript 字符串等不同上下文分别处理。
- 避免直接使用 `innerHTML`、`document.write` 等危险 API。
- 必须渲染富文本时，使用可信白名单清洗库。
- 开启 CSP，限制脚本来源和内联脚本执行。
- Cookie 设置 `HttpOnly`，降低脚本窃取 Cookie 的风险。
- 前端框架中不要绕过默认转义能力。

React 默认会转义文本内容，所以这类写法相对安全：

```jsx
<div>{userInput}</div>
```

但下面这种写法需要非常谨慎：

```jsx
<div dangerouslySetInnerHTML={{ __html: userInput }} />
```

如果 `userInput` 没经过严格清洗，就可能引入 XSS。

### 4. CSRF 是什么

CSRF，全称 Cross-Site Request Forgery，中文常叫跨站请求伪造。它的核心是：攻击者诱导已登录用户访问恶意页面，由浏览器自动携带目标站点的 Cookie，向目标站点发起请求。

CSRF 成立通常需要三个条件：

- 用户已经登录目标网站。
- 登录态依赖浏览器自动携带的 Cookie。
- 目标接口没有验证请求是否真的来自可信页面或用户意愿。

典型链路：

```text
用户登录 A 网站
-> 浏览器保存 A 网站 Cookie
-> 用户访问攻击者 B 网站
-> B 网站构造请求发往 A 网站
-> 浏览器自动带上 A 网站 Cookie
-> A 网站误以为是用户本人操作
```

### 5. CSRF 的常见攻击方式

如果某个转账接口支持 GET 请求：

```html
<img src="https://bank.example/transfer?to=attacker&amount=1000" />
```

用户访问恶意页面时，浏览器加载图片就会发起请求。只要 Cookie 自动携带，服务端又没有额外验证，就可能产生风险。

即使接口是 POST，也可以通过自动提交表单诱导请求：

```html
<form action="https://example.com/update-email" method="POST">
  <input name="email" value="attacker@example.com" />
</form>
<script>
  document.forms[0].submit();
</script>
```

所以 CSRF 的重点不是请求方法，而是服务端有没有验证请求来源和用户意愿。

### 6. CSRF 的防护思路

CSRF 防护的核心是：**不能只依赖 Cookie 判断用户身份，还要验证请求确实由可信页面发起。**

常见措施：

- 使用 CSRF Token。
- Cookie 设置 `SameSite=Lax` 或 `SameSite=Strict`。
- 对关键操作验证 `Origin` 和 `Referer`。
- 关键接口只接受非简单请求，并校验自定义请求头。
- 转账、改密、删除等高风险操作增加二次确认。
- GET 请求只用于读取，不用于产生副作用。

CSRF Token 的典型逻辑：

```text
服务端生成 token
-> 页面表单或接口请求携带 token
-> 服务端校验 token
-> token 不存在或不匹配则拒绝
```

攻击者网站无法读取目标站点页面里的 token，也就很难伪造合法请求。

### 7. XSS 和 CSRF 的区别

可以用一句话记住：

```text
XSS 是让恶意脚本在用户页面里执行。
CSRF 是借用户 Cookie 发起伪造请求。
```

更具体一点：

| 对比项 | XSS | CSRF |
| --- | --- | --- |
| 攻击目标 | 浏览器页面执行环境 | 用户登录态和接口请求 |
| 关键条件 | 页面存在脚本注入点 | 接口只依赖 Cookie 鉴权 |
| 典型后果 | 盗取信息、篡改页面、劫持操作 | 伪造转账、改密、提交表单 |
| 主要防护 | 转义、清洗、CSP、HttpOnly | CSRF Token、SameSite、Origin 校验 |
| 前端关注点 | 不危险渲染用户输入 | 请求携带 token，避免副作用 GET |

### 8. 两者的关系

XSS 和 CSRF 不是互斥关系。很多时候 XSS 的危害更大，因为一旦攻击者能执行脚本，就可以读取页面数据、发起接口请求、绕过部分 CSRF 防护。

例如，如果 CSRF Token 暴露在 DOM 中，而页面又存在 XSS，攻击脚本就可能先读取 token，再带着 token 发起请求。

所以安全优先级上，通常要先确保没有 XSS，再完善 CSRF 防护。

### 9. 前端项目里的落地清单

开发时可以按这份清单检查：

- 所有用户输入默认不可信。
- 渲染文本时使用框架默认转义能力。
- 富文本必须经过白名单清洗。
- 禁止把 URL 参数直接写入 `innerHTML`。
- Cookie 根据用途设置 `HttpOnly`、`Secure`、`SameSite`。
- 写操作接口使用 POST/PUT/DELETE，不用 GET 做副作用。
- 请求头或表单中携带 CSRF Token。
- 服务端校验 `Origin` / `Referer`。
- 高风险操作增加验证码、二次确认或重新认证。
- CSP 策略逐步上线，先 Report-Only 再强制。

### 10. 总结

XSS 和 CSRF 都是 Web 安全基础题，但真正落地时经常出问题。

XSS 防的是“恶意脚本执行”，重点在输入输出处理、富文本清洗和 CSP。

CSRF 防的是“伪造用户请求”，重点在 token、SameSite、来源校验和接口语义。

前端开发不能只依赖框架默认安全能力。框架能帮你转义普通文本，但一旦使用富文本、第三方脚本、Cookie 登录态、跨站请求，就必须把安全边界设计清楚。
