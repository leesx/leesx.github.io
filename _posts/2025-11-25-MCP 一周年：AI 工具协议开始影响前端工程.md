# MCP 一周年：AI 工具协议开始影响前端工程

2025 年 11 月，Model Context Protocol 发布一周年回顾和新规格。MCP 原本是为模型连接工具和数据源而生的协议，但它已经开始影响前端开发工作流。

前端工程里的 AI 工具正在从“聊天窗口”变成“能理解项目的开发代理”。它需要读取代码、查看浏览器错误、运行测试、访问设计稿、查询文档、调用部署平台。没有统一协议时，每个工具都要单独接入这些能力，权限和上下文也难管理。

MCP 的价值在于定义客户端、服务器、工具、资源和授权之间的交互方式。对前端团队来说，典型场景包括：

- IDE 里的 AI 读取项目结构和运行脚本。
- 浏览器调试工具把 console、network、route 信息暴露给 AI。
- 设计系统、CMS、监控平台通过 MCP 给开发代理提供上下文。
- 内部工具以标准方式暴露查询、生成、修改能力。

2025 年的 MCP 规格重点加强了授权、安全、结构化输出和任务式工作流。尤其是 OAuth 2.1、PKCE、resource 参数、token audience validation 这些要求，说明 MCP 不只是本地玩具协议，而是在向企业级集成靠拢。

### 前端团队应该怎么理解

不要急着把所有东西都做成 MCP Server。先找上下文最稳定、收益最大的工具：组件文档、设计 token、错误查询、部署日志、接口 mock。每个 MCP Server 都应该有清楚的权限边界和审计策略。

未来的前端工程师可能不仅写页面，也要维护“AI 能理解项目的接口”。文档、脚本、日志、测试和设计资产都会变成 AI 工具链的一部分。
