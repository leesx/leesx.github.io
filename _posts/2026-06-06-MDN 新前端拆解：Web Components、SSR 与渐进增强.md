# MDN 新前端拆解：Web Components、SSR 与渐进增强

## 背景

MDN Blog 在 2026-04-08 发布了 `Under the hood of MDN's new frontend`。这不是一篇“网站改版复盘”层面的内容，而是一篇非常值得前端团队细读的工程说明：**当一个内容型站点既有大规模静态内容、复杂文档结构、多语言支持，又需要少量但关键的交互能力时，前端架构应该怎么重做。**

MDN 把自己的发布链路讲得很清楚：

- 文档内容以 Markdown 维护
- 构建工具先把内容转成 HTML 和附带元数据的 JSON
- 前端再遍历这些 JSON 做 SSR，产出完整页面
- 最终把 HTML、CSS、JavaScript 分发到全球 CDN

更关键的是，MDN 解释了他们为什么要重建前端：旧站基于 React，但内容主体本身是静态文档，React 实际上更像包在外层的壳。结果是：

- UI 修改越来越容易引入技术债
- CSS 作用域混乱，组件之间互相污染
- 需要给用户下发一大块渲染阻塞 CSS
- 文档中的交互能力被迫长期混用 React 和原生 DOM 操作

## 为什么重要

- **这是一份很少见的一手工程样本**：很多团队都在做“内容站 + 局部交互”，但官方很少像 MDN 这样把问题、约束和取舍讲透。
- **它提醒我们不要机械地把 SPA 当成默认答案**：当内容主体本来就是静态的，强行用客户端框架包住全部页面，往往会让维护成本和前端负担持续上升。
- **Web Components 正在重新回到主流工程语境**：MDN 没有把它当概念秀，而是把它用于“在静态内容里植入有限交互”的真实场景。

## 前端/AI 开发落地建议

### 1. 对内容型站点，先判断“页面主体是谁”

如果你的页面主要是文档、教程、博客、知识库、帮助中心，而不是高度交互的业务面板，建议先问一个问题：

- 页面主体是数据驱动交互，还是内容分发？

如果答案偏后者，就应该优先考虑：

- 让内容产物天然可 SSR / 可静态化
- 把交互能力收敛成局部组件
- 避免让整个页面生命周期都被客户端框架接管

这比“整站 React 化”更容易得到长期可维护的结果。

### 2. 把 Web Components 当成内容内嵌交互层

MDN 选择用 Lit 和 Web Components 来承载文档中的交互模块，例如延迟加载的可交互教学组件。这种思路对很多站点都适用：

- 代码示例的复制、运行、展开
- 嵌入式 demo、沙箱或播放器
- 文档中的交互式提示、反馈、配置面板

这样做的核心价值不是“技术更新”，而是让交互组件直接嵌进内容结构，不需要再为这类小能力引入整页级 hydration 成本。

### 3. 为 AI 生成内容和 Agent 改文档预留更稳的架构边界

随着 AI 写文档、生成教程、自动补示例、批量改知识库越来越常见，内容站会越来越依赖“内容层”和“交互层”的清晰分离。建议：

- 让 Markdown / HTML 成为稳定内容源
- 把交互增强做成边界清楚的组件
- 让 AI 只改内容数据，不轻易碰整体页面运行时

否则一旦把内容、样式、交互、构建脚本揉成一团，后续无论人工维护还是 agent 自动修改，都会很脆弱。

## 注意事项

- **不要把这篇文章误读成“React 不适合内容站”**：MDN 讲的是它自己的架构约束，不是对所有 React 站点下结论。
- **Web Components 也不是零成本方案**：仍然要处理样式策略、浏览器能力、SSR 配合和团队心智模型。
- **渐进增强仍然是关键**：MDN 明确提到会把较新的平台能力作为增强手段，并保留旧环境 fallback，这一点对公共内容站尤其重要。
