# MCP 工具注解：风险提示不等于权限控制

## 背景

Model Context Protocol 官方博客在 2026-03-16 发布了 `Tool Annotations as Risk Vocabulary: What Hints Can and Can't Do`。这篇文章讨论的重点不是“再加几个字段”，而是一个更容易被 AI 工程团队误解的问题：

**工具注解可以帮助表达风险语义，但它不是权限系统，也不能替代真正的审批边界。**

官方在摘要里明确提到，MCP 的 tool annotations 用来描述工具的行为特征，例如是否只读、是否有破坏性、是否幂等、是否会访问本地环境之外的资源。随着社区提出越来越多新的注解提案，维护者开始主动划清边界：哪些能力适合放进 annotations，哪些不该指望它解决。

## 为什么重要

- **很多团队正在把“工具描述”误当成“安全机制”**：如果一个工具写了只读或 destructive，并不代表客户端一定会安全地执行它；这些字段本质上更接近风险提示词汇表。
- **AI 工作流的真正风险不只在工具名上**：同一个工具在不同参数、不同上下文、不同审批链里，风险可能完全不同。官方这次专门强调“risk vocabulary”，其实是在提醒大家不要过度静态化理解工具风险。
- **这会直接影响前端里的 agent UI 设计**：当产品开始把 MCP 工具接进聊天界面、控制台或工作流面板时，前端不该只展示一个“只读 / 可写”的标签，而要把用户真正关心的副作用暴露出来。

## 前端/AI 开发落地建议

### 1. 把 annotations 当作提示层，不要当作授权层

工程上更稳妥的做法是把 tool annotations 放在这几类用途里：

- 给用户展示风险提示
- 给模型提供粗粒度决策线索
- 给日志和审计系统补充行为标签

但真正决定能不能执行，仍然应该由：

- host 的审批策略
- 运行时权限隔离
- 明确的用户确认步骤

### 2. UI 要展示“会发生什么”，而不只是“工具类型”

如果前端已经在做 agent 操作面板，建议把风险展示做成更具体的说明，例如：

- 是否修改文件
- 是否请求网络
- 是否会删除或覆盖资源
- 是否可能重复执行

这样比单纯显示注解字段更有工程价值，也更符合官方把 annotations 视为“风险词汇表”的思路。

### 3. 审批模型要围绕副作用，而不是围绕命名

同一个 `writeFile` 工具，写临时草稿和覆盖生产配置文件的风险完全不同。实践上应该把审批粒度尽量压到“动作 + 目标 + 参数”这一层，而不是只看工具本身的标签。

## 注意事项

- **不要把 annotations 设计成越来越像 ACL**：官方这篇文章本质上是在给社区降温，说明注解不是万能控制面。
- **静态标签无法覆盖动态风险**：真实风险往往取决于输入参数、外部系统状态和用户当前会话上下文。
- **模型可以参考 annotations，但不应独立承担安全责任**：审批、隔离、审计仍然要由宿主环境兜底。
