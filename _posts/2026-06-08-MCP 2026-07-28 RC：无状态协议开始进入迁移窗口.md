# MCP 2026-07-28 RC：无状态协议开始进入迁移窗口

## 背景

Model Context Protocol 官方博客在 2026-05-21 发布了 `The 2026-07-28 MCP Specification Release Candidate`。如果说 3 月那篇路线图是在解释方向，这次 RC 则是在给出真正需要开发者开始迁移和验证的版本边界。

官方直接说明，这是自 MCP 发布以来最大的一次修订，核心变化包括：

- 协议核心转向无状态
- 扩展体系正式化
- `MCP Apps` 进入规范语义
- `Tasks` 从实验性核心能力转为扩展
- 授权进一步向 OAuth / OpenID Connect 的真实部署靠拢

## 为什么重要

- **这不是概念更新，而是带破坏性变化的迁移窗口**：官方明确写了，这个 RC 含有 breaking changes，最终规范将在 2026-07-28 发布。也就是说，SDK、client、server 的实现者已经进入验证期，而不是继续观望。
- **“无状态”会直接改变部署方式**：RC 最核心的变化是移除协议层会话，把 MCP 变成更适合普通 HTTP 基础设施的形态。这会影响负载均衡、缓存、追踪、任务恢复以及 server 设计习惯。
- **MCP Apps 和 Tasks 开始更像可落地产品能力**：一个负责把交互式 UI 作为服务器可声明资源交给 host 渲染，一个负责长任务生命周期。对前端 Agent 产品来说，这两件事都非常接近真实业务。

## 前端/AI 开发落地建议

### 1. 不要再把 session 绑死在协议层

官方说明，协议层会话和握手已经被移除，但这不代表你的业务必须无状态。更合理的做法是：

- 需要状态时，由 tool 显式返回业务句柄
- 后续调用通过普通参数传回这个句柄
- 把状态管理看成应用层职责，而不是协议层魔法

这会让 MCP server 更容易水平扩展，也更适合接入标准 HTTP 网关体系。

### 2. 如果你在做交互式工具，开始按 MCP Apps 的思路设计

RC 对 `MCP Apps` 的定义很关键：server 可以声明交互式 HTML 界面，由 host 在沙箱 iframe 中渲染，并通过同一套 JSON-RPC 协议回到 host。对前端工程意味着：

- 工具界面不再只是“工具调用后的附件”
- host 可以在运行前预取、缓存和安全审查 UI 模板
- UI 发起的操作仍走审计和审批链路

这比“让 agent 返回一段自由 HTML”工程化得多。

### 3. 长任务实现要提前适配新的 Tasks 生命周期

官方明确写到，`Tasks extension` 下服务端可以在 `tools/call` 后返回 task handle，客户端再用 `tasks/get`、`tasks/update`、`tasks/cancel` 驱动任务，并且 `tasks/list` 被移除。工程上建议：

- 不要继续依赖旧的 experimental Tasks 语义
- 把任务轮询、取消、输入补全设计成显式状态机
- 为 client 和 server 预留扩展能力协商逻辑

## 注意事项

- **最终版日期不是“未来某个时候”，而是明确的 2026-07-28**：RC 从 2026-05-21 起锁定，官方给了大约十周验证窗口。
- **无状态不等于无上下文**：协议层去 session，不代表应用层不能保存购物篮、浏览器实例、工作目录等业务状态。
- **旧版 experimental Tasks 用户需要迁移**：官方已经明确指出，原有生命周期不再兼容。
