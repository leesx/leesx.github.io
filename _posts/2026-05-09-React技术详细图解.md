# React 技术详细图解

![React 技术详细图解](/img/react-technical-diagram.png)

这张图把 React 的核心知识拆成了几个层次：UI 描述、状态驱动、事件系统、Fiber 渲染、DOM 提交、副作用、生态工具。理解这些模块之间的关系，比单独记 API 更重要。

React 的核心思想可以概括为一句话：**数据自上而下流动，视图由数据映射而来，事件触发状态变化，状态变化驱动重新渲染。**

### 1. React 核心概念（UI 层）

React 是一个用于构建用户界面的 JavaScript 库，适合用组件化、声明式的方式开发复杂 Web 应用。

常见核心模块包括：

- JSX 与组件：用类似 HTML 的语法声明 UI 结构，本质会被编译为 JavaScript 调用。
- Props / State：Props 负责父子组件之间的数据传递，State 负责组件内部可变状态。
- Hooks：让函数组件拥有状态、副作用、上下文、缓存等能力。
- Context：跨组件共享数据，适合主题、语言、登录态等全局或半全局信息。
- 事件系统：React 对浏览器事件做了一层封装，统一事件处理模型。

一个最简单的状态组件：

```js
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}
```

这里的关键不是 `button`，而是 `setCount`。事件触发状态更新，状态更新进入 React 调度流程，然后计算新的 UI。

### 2. 事件向上冒泡

React 的事件处理遵循类似 DOM 的冒泡模型。用户点击按钮后，事件会从目标节点向上触发，经过 React 合成事件系统，再调用组件中注册的事件处理函数。

一个典型链路是：

```text
DOM 事件触发 -> React 合成事件系统 -> 组件事件处理函数 -> setState / dispatch
```

如果事件处理函数调用了 `setState` 或状态管理库的 `dispatch`，React 就会把这次更新放入更新队列，等待调度。

### 3. 状态变化触发渲染

React 中不要直接修改视图，而是修改状态。

状态变化之后，React 会做几件事：

```text
setState / dispatch
-> 状态更新入队列
-> 调度更新
-> 触发重新渲染
```

这就是 React 的“声明式”本质。开发者描述 UI 应该长什么样，React 负责在数据变化后计算最小必要更新。

### 4. 渲染流程：Fiber 架构

React 早期的更新过程更接近递归遍历，一旦开始渲染就很难中断。Fiber 架构把渲染工作拆成一个个更小的任务单元，让 React 可以中断、恢复、复用和调整优先级。

图中渲染流程可以拆成四个概念：

- Virtual DOM：用 JavaScript 对象描述 UI 结构。
- Fiber 架构：把组件树拆成可调度的工作单元。
- Reconciliation：比较新旧虚拟 DOM，找出需要更新的部分。
- Scheduler：根据任务优先级管理执行顺序和时间切片。

简化理解：

```text
状态变化
-> 生成新的 Virtual DOM
-> Fiber 遍历组件树
-> Diff 新旧结构
-> 标记需要更新的节点
```

### 5. Render 阶段和 Commit 阶段

React 更新可以分成两个阶段。

#### Render 阶段（可中断）

Render 阶段主要做计算，不直接操作真实 DOM。

这一阶段会：

- 构建 Fiber 树。
- 计算差异。
- 准备更新列表。

因为它不直接改页面，所以可以被中断、暂停或重新开始。Concurrent React 的很多能力都建立在这个基础上。

#### Commit 阶段（不可中断）

Commit 阶段负责把计算结果真正应用到页面。

这一阶段会：

- 执行 DOM mutation。
- 处理布局相关逻辑。
- 执行副作用。

Commit 阶段一旦开始就必须尽快完成，因为用户页面会在这里发生真实变化。

### 6. DOM 更新

React 不会每次状态变化都粗暴重建整个页面，而是根据 Diff 结果把必要变更应用到真实 DOM。

流程可以理解为：

```text
计算差异 -> 标记更新 -> Commit 阶段 -> 修改真实 DOM -> 页面完成更新
```

这也是为什么 React 性能优化不应该只盯着“少渲染”。更准确的问题是：

- 哪些状态变化影响范围太大？
- 哪些组件树太深或太重？
- 哪些计算应该缓存？
- 哪些副作用不该跟渲染耦合？

### 7. 副作用执行：useEffect 和 useLayoutEffect

React 的渲染应该保持纯粹。请求接口、订阅事件、操作 DOM、写日志、定时器这些都属于副作用。

常见写法：

```js
function Home() {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch("/api/user")
      .then((res) => res.json())
      .then(setUser);
  }, []);

  return <div>{user?.name}</div>;
}
```

`useEffect` 和 `useLayoutEffect` 的区别：

- `useEffect`：浏览器完成绘制后异步执行，不阻塞渲染，适合请求、订阅、日志等。
- `useLayoutEffect`：DOM 更新后、浏览器绘制前同步执行，适合读取布局或同步修改布局。

实际项目里应优先使用 `useEffect`。只有确实需要在绘制前读取或修正布局时，再使用 `useLayoutEffect`。

### 8. 自定义 Hook

自定义 Hook 的价值是复用状态逻辑，而不是复用 UI。

例如封装一个 toggle：

```js
function useToggle(initial = false) {
  const [state, setState] = useState(initial);
  const toggle = useCallback(() => {
    setState((s) => !s);
  }, []);

  return [state, toggle];
}
```

组件中只关心“当前状态”和“切换动作”，不用关心内部怎么实现。

### 9. 状态管理

React 自带 `useState`、`useReducer`、`Context`，已经能覆盖很多场景。只有当状态跨页面、跨模块、跨业务流程共享时，才需要引入 Redux、Zustand、Recoil 等状态管理库。

状态管理选型可以按复杂度判断：

- 局部状态：`useState`。
- 复杂局部状态：`useReducer`。
- 中低频全局状态：`Context`。
- 高频共享状态或复杂业务流：Redux Toolkit、Zustand 等。

不要把所有状态都放进全局 store。状态离使用位置越近，组件越容易维护。

### 10. 路由管理

React Router 负责声明式路由、嵌套路由、路由守卫和导航。

常见 API：

- `useNavigate`：编程式跳转。
- `useParams`：读取路由参数。
- `Outlet`：渲染嵌套路由出口。
- loader / action：在新版本数据路由中处理数据加载和提交。

路由不只是 URL 映射组件，还承担页面级数据边界、权限边界和缓存边界。

### 11. 服务端渲染：SSR / SSG

React 可以通过 Next.js 等框架实现 SSR、SSG、ISR、React Server Components 等能力。

它们解决的问题主要是：

- 首屏加载性能。
- SEO。
- 服务端数据预取。
- 静态内容生成。
- 客户端包体积控制。

不过 SSR 也会带来复杂度，比如 hydration mismatch、服务端和客户端环境差异、缓存失效、部署平台差异。需要根据业务场景选择，而不是默认所有页面都 SSR。

### 12. 性能优化

React 性能优化应该从数据流和组件边界开始，而不是一上来就写 `memo`。

常见工具：

- `React.memo`：避免 props 没变时重复渲染组件。
- `useMemo`：缓存昂贵计算结果。
- `useCallback`：缓存函数引用。
- `React.lazy`：组件懒加载。
- `Suspense`：声明异步加载状态。
- React DevTools Profiler：定位真实性能瓶颈。

优化顺序建议：

1. 先用 Profiler 找到慢在哪里。
2. 缩小状态影响范围。
3. 拆分过重组件。
4. 延迟加载不在首屏的模块。
5. 最后再考虑 memo 化。

### 13. 生态与工具

React 项目通常离不开这些工具：

- Vite：开发服务器和构建工具。
- TypeScript：类型系统。
- ESLint：代码规范和 Hooks 规则检查。
- Jest / Vitest：单元测试。
- React Testing Library：组件测试。
- Playwright：端到端测试。
- React DevTools：组件树和性能分析。

这些工具的目的不是堆技术栈，而是让项目在长期维护中保持可读、可测、可调试。

### 14. 常见触发更新的场景

图中列出的典型更新来源包括：

- `setState` 或 `dispatch`。
- 父组件重新渲染。
- Context 值变化。
- Props 变化。
- `useEffect` 中更新状态。

其中最容易被忽略的是“父组件重新渲染”。即使子组件 props 看起来没变，只要父组件渲染，子组件默认也会重新执行函数体。是否造成性能问题，要看组件本身是否昂贵，不要过早优化。

### 15. 最佳实践

- 组件设计：高内聚、可复用、单一职责。
- 状态管理：尽量保持局部，避免无意义全局化。
- 性能优化：按证据优化，避免不必要的重新渲染。
- 副作用处理：正确使用 `useEffect` 和 `useLayoutEffect`。
- 代码质量：TypeScript、ESLint、单元测试一起使用。
- 组件边界：页面组件负责组装，业务组件负责逻辑，基础组件负责展示。

### 总结

React 的整体流程可以串成一条线：

```text
用户交互
-> 事件处理
-> 更新状态
-> 调度渲染
-> 计算差异
-> Render 阶段
-> Commit 阶段
-> 更新 DOM
-> 执行副作用
-> 页面完成更新
```

理解这条线后，再看 Hooks、状态管理、路由、SSR、性能优化，就不会只是零散 API，而是一套围绕“状态驱动 UI”的完整体系。
